# IdP Vendor Coverage Matrix

Research date: 2026-05-28

Scope: this matrix compares `tigrbl_auth` with Auth0, Okta, Keycloak, FusionAuth, and Amazon Cognito for Identity Provider coverage. "IdP" has two meanings that should stay separate:

1. A first-party identity provider or OIDC Provider that authenticates users and issues tokens for applications.
2. An identity broker or federation surface that lets users authenticate through upstream social, enterprise, SAML, OIDC, LDAP, or other external providers.

`tigrbl_auth` already targets the first meaning through its public OAuth/OIDC issuer. The second meaning is a trust-plane/federation product lane and is much thinner today.

## Interpretation

The competitive pattern is clear: mature identity vendors treat external IdPs as their own product surface, not as incidental OIDC behavior. The product usually includes:

| Product concern | What mature vendors expose |
| --- | --- |
| Provider inventory | Create, list, update, delete, enable/disable upstream identity providers. |
| Protocol families | Social OAuth/OIDC, enterprise OIDC, SAML, sometimes LDAP/directory connectors and external JWT. |
| Routing | Home realm discovery, provider buttons, domain hints, organization routing, app-specific provider selection. |
| Identity resolution | Account linking, subject aliases, duplicate detection, and safe unlinking. |
| Provisioning | JIT user creation, attribute mapping, profile sync, group sync, and activation policy. |
| Security | Issuer pinning, metadata refresh, signing/encryption requirements, key rotation, callback validation, and credential storage. |
| Operations | Audit logs, event hooks, failed login diagnostics, drift detection, and provider health. |

For `tigrbl_auth`, the clean architecture is:

| Layer | Recommended owner |
| --- | --- |
| First-party OIDC Provider behavior | `tigrbl-identity-oidc`, `tigrbl-auth-api-public` |
| External IdP / federation domain models | Future `tigrbl-identity-federation` or a clearly named federation module under admin/storage before package split |
| Provider credentials and secret material | `tigrbl-identity-credentials` plus `tigrbl-identity-storage` |
| Subject aliasing and account linking | `tigrbl-identity-principals` plus canonical storage |
| Tenant/app/provider policy | `tigrbl-identity-policy`, tenant-admin/developer frontdoors |
| Provider configuration surfaces | `tigrbl-auth-api-tenant-admin`, `tigrbl-auth-api-developer`, and their UIX apps |
| Hosted provider selection | `@tigrbl-auth/public-uix` over public issuer behavior |

Recommended near-term stance:

1. Keep claiming current support for first-party OIDC Provider behavior.
2. Do not claim social login, enterprise SSO, SAML federation, LDAP federation, or IdP brokering as implemented until there are tables, routes, tests, and product docs.
3. Promote `SubjectAlias` from a principal concept into canonical storage and tests before implementing account linking.
4. Add a canonical `IdentityProvider` model only after deciding provider scoping: platform-wide, tenant-wide, app-specific, or organization-specific.
5. Treat IdP federation as a trust-plane product adjacent to SCIM, M2M/workload federation, and cross-cloud trust.

# IAM, CIAM, PAM Vendor Coverage Matrix

Research date: 2026-05-28

Scope: this matrix compares `tigrbl_auth` with Auth0, Okta, Keycloak, FusionAuth, Amazon/AWS, and Microsoft Entra across IAM, Workforce IAM, CIAM, PAM, IGA, non-human/workload identity, developer identity platform, and management API surfaces.

These are product categories, not protocols. OAuth, OIDC, MFA, SCIM, RBAC, IdP federation, M2M, and policy are ingredients used to assemble IAM/CIAM/PAM/IGA products.

## Interpretation

The vendor pattern is product-family segmentation:

| Product category | Vendor pattern |
| --- | --- |
| CIAM | Hosted login, registration, customer profile, organizations, social/enterprise IdPs, MFA, consent, app/client integration, APIs. |
| Workforce IAM | Workforce directory, SSO, app assignments, lifecycle, device/risk policy, admin roles, governance hooks. |
| PAM | JIT privileged access, approval, vaulting, session brokering, workload/admin privilege, break-glass, recording/audit. |
| IGA | Access requests, entitlement catalog, access reviews, certifications, SoD, compliance workflows. |
| Workload identity | Service principals, workload federation, M2M credentials, token exchange, key rotation, resource validation. |
| Developer identity platform | App registration, SDKs, integration docs, test clients, discovery/JWKS, resource-server tooling. |

For `tigrbl_auth`, the strongest near-term product framing is:

```text
CIAM-oriented identity provider
+ Management API
+ Developer Platform
+ Workload Identity lane
```

The repo should use "IAM foundations" or "identity platform" for the broad architectural foundation, and avoid claiming PAM or IGA until those categories have concrete runtime products, APIs, UI, tests, and SSOT contracts.

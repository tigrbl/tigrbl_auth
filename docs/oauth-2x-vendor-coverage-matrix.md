## Interpretation

The competitive pattern is not "OAuth 2.1 as a separate product." The pattern is a broad OAuth 2.0/OIDC authorization-server product plus stricter security posture exposed through app/client policy, third-party application controls, API access management, DPoP/PAR support, and M2M/service-client products.

For `tigrbl_auth`, the strongest next move is an OAuth 2.1 alignment profile over the existing surfaces:

- `tigrbl-auth-api-public` remains the OAuth/OIDC authorization-server frontdoor.
- `tigrbl-auth-api-developer` owns client/app registration policy and should enforce OAuth 2.1-safe defaults.
- `tigrbl-auth-api-service-admin` owns M2M/workload identity lifecycle around client credentials.
- `tigrbl-auth-api-resource-validation` owns resource-server metadata, JWKS, introspection, and token validation integration.
- `tigrbl-auth-protocol-rp` and `@tigrbl-auth/rp` own client-side OAuth/OIDC behavior, including PKCE defaults.

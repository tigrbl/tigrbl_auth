# OIDC Party and Subsystem Vendor Coverage Matrix

Research date: 2026-05-27

This matrix maps OpenID Connect and inherited OAuth parties/subsystems to `tigrbl_auth` and five identity competitors. Rows are OIDC/OAuth actors or concrete protocol subsystems; columns are vendors. "Coverage" describes where the vendor exposes the capability: provider/API, relying-party SDK, resource-server verifier, hosted UI, admin/control plane, or known product gap.

Normative anchors: OIDC Core defines End-User, Relying Party, OpenID Provider, ID Token, UserInfo, and related flows. OAuth 2.0 contributes the Resource Owner, Client, Authorization Server, and Resource Server roles. OIDC Discovery and logout specifications define provider metadata and logout/session surfaces.

| OIDC / OAuth party or subsystem | `tigrbl_auth` | Auth0 | Okta | Keycloak | FusionAuth | Amazon Cognito |
|---|---|---|---|---|---|---|
| End-User / Subject / Resource Owner | User principals exist in the identity model and are surfaced through [`public-api`](../pkgs/70-apis/tigrbl-auth-api-public/README.md) login and admin APIs; first-class end-user account self-service remains underframed. | End-users authenticate through [Universal Login](https://auth0.com/docs/authenticate/login/auth0-universal-login); self-service security/account operations appear in [My Account API](https://auth0.com/docs/api/myaccount/). | Users are first-class management objects in [Users API](https://developer.okta.com/docs/api/openapi/okta-management/management/tag/User/); end-user operations appear in [MyAccount API](https://developer.okta.com/docs/api/openapi/okta-myaccount/myaccount/overview/) and End-User Dashboard. | Realm users are managed in Admin REST/Admin Console; users self-manage profile, password, sessions, and consents through [Account Console](https://www.keycloak.org/docs/latest/server_admin/#account-console). | Users are first-class in [User API](https://fusionauth.io/docs/apis/users), registrations, hosted login, passwordless, MFA, and WebAuthn APIs. | User pool users are first-class subjects in [Cognito User Pools](https://docs.aws.amazon.com/cognito/latest/developerguide/cognito-user-pools.html); self-service flows are exposed through Managed Login, hosted UI, Amplify, and runtime auth APIs. |
| User Agent / Browser | Browser-side surface is [`@tigrbl-auth/public-uix`](../apps/public-uix/README.md) plus [`@tigrbl-auth/rp`](../apps/rp/README.md); OIDC adapter code exists in public UIX services. | Browser is mediated by Universal Login, SPA SDKs, and redirect flows in [Authentication API](https://auth0.com/docs/api/authentication). | Browser mediation is via Okta-hosted sign-in and [Sign-In Widget](https://developer.okta.com/docs/concepts/sign-in-widget/) plus Auth JS/framework SDKs. | Browser participates through Keycloak login pages/themes and the [JavaScript adapter](https://www.keycloak.org/securing-apps/javascript-adapter). | Browser flows are handled by hosted login pages and React/Angular/Vue SDKs. | Browser flows are handled by [Managed Login](https://docs.aws.amazon.com/cognito/latest/developerguide/cognito-user-pools-managed-login.html), classic hosted UI, and Amplify UI Authenticator. |
| Relying Party (RP) / OAuth Client | Consumer RP package exists as [`tigrbl-auth-protocol-rp`](../pkgs/40-protocols/tigrbl-auth-protocol-rp/README.md), and browser/client package exists as [`@tigrbl-auth/rp`](../apps/rp/README.md); coverage includes discovery, UserInfo integration, and RP-initiated logout helpers. | Covered by SPA, mobile, server SDKs and application/client configuration through [Applications](https://auth0.com/docs/get-started/applications). | Covered by Auth JS, framework SDKs, mobile SDKs, and app integration models through [Applications API](https://developer.okta.com/docs/api/openapi/okta-management/management/tag/Application/). | Covered by JavaScript adapter, Java Admin Client for client setup, generic OIDC client support, and client configuration in Admin REST. | Covered by application objects, hosted login, and framework SDKs; applications model OAuth clients in [Application API](https://fusionauth.io/docs/apis/applications). | Covered by app clients in user pools, Amplify Auth, and OIDC-compatible app integrations. |
| OpenID Provider (OP) / Authorization Server (AS) | Provider-side surface is [`tigrbl-auth-api-public`](../pkgs/70-apis/tigrbl-auth-api-public/README.md), backed by identity OAuth/OIDC packages and server/runtime assembly. | Auth0 tenant acts as OP/AS through [Authentication API](https://auth0.com/docs/api/authentication) and tenant issuer. | Okta org/custom authorization server acts as OP/AS through [OIDC and OAuth API](https://developer.okta.com/docs/reference/api/oidc/) and API Access Management. | Each realm acts as OP/AS through [realm OIDC endpoints](https://www.keycloak.org/securing-apps/oidc-layers). | FusionAuth application/tenant stack acts as OP/AS through OAuth endpoints and JWT/Login APIs. | Cognito user pool acts as OP/AS through [OAuth 2.0 endpoints](https://docs.aws.amazon.com/cognito/latest/developerguide/cognito-userpools-server-contract-reference.html). |
| Provider discovery / issuer metadata | [`public-api`](../pkgs/70-apis/tigrbl-auth-api-public/README.md) and [`resource-validation-api`](../pkgs/70-apis/tigrbl-auth-api-resource-validation/README.md) expose OIDC discovery and protected-resource metadata. | Discovery and issuer behavior documented under Authentication API and tenant metadata/JWKS pages. | Discovery exposed through authorization-server metadata in [OIDC reference](https://developer.okta.com/docs/reference/api/oidc/). | Discovery exposed in realm OIDC endpoint layer. | OAuth/OIDC endpoint metadata and JWKS are part of OAuth/JWT documentation. | User-pool OIDC endpoints include issuer metadata and JWKS endpoint references. |
| Authorization endpoint / user authorization ceremony | `public-api` exposes `/authorize`; hosted-login UIX is [`public-uix`](../apps/public-uix/README.md). | `/authorize` with Universal Login, Organizations, social/enterprise federation, MFA, consent and prompt behavior. | `/authorize` via Okta-hosted sign-in/sign-in widget and authorization server policies. | Realm authorization endpoint plus login themes/authentication flows. | OAuth authorize endpoint plus hosted login pages and theme customization. | `/oauth2/authorize` through Managed Login/hosted UI with IdP selection and configured scopes. |
| Token endpoint / token service | `public-api` exposes `/token`; token behavior is backed by OAuth/credentials/JOSE packages. | `/oauth/token` handles authorization-code, refresh token, client credentials/M2M, and related grants. | Token endpoint supports configured grant types, custom authorization servers, service apps, and client credentials. | Realm token endpoint handles OIDC/OAuth grants including service-account/client credentials. | OAuth token endpoint plus JWT APIs support grants, refresh tokens, and client credentials. | `/oauth2/token` supports auth-code, refresh, and client credentials where configured for app clients/resource servers. |
| ID Token / claims issuer | `tigrbl-identity-jose`, `tigrbl-auth-protocol-oidc`, and `public-api` cover JWT/JWK/JWKS and OIDC behavior; current proof coverage is stronger on provider issuance than on advanced claim customization product surfaces. | ID tokens and custom claims configured through Auth0 Actions/rules-equivalent behavior and Management API resources. | ID tokens, claims, scopes, policies, and authorization servers are configured through API Access Management and Authorization Servers API. | ID tokens issued by realm OP; mappers, scopes, roles, and client scopes shape claims. | JWT and OAuth APIs issue ID/access tokens; lambdas and application config customize claims. | User pool tokens include ID/access/refresh tokens; pre token generation Lambda customizes claims. |
| JWKS / signing-key subsystem | `public-api` and `resource-validation-api` expose JWKS; key material is owned by `tigrbl-identity-jose`; platform admin has signing/key rotation admin surfaces. | JWKS and signing keys exposed by tenant issuer; key lifecycle managed through Management API keys/signing-key surfaces. | JWKS published per authorization server; key rotation and Authorization Servers API cover key lifecycle. | Realm keys and certs/JWKS endpoints; admin console/API manages realm keys. | JWT API and Key API cover signing keys and JWKS-style verification materials. | JWKS endpoint per user pool; key rotation is AWS-managed for user-pool signing keys. |
| UserInfo endpoint / profile claims resource | `public-api` exposes `/userinfo`; `tigrbl-auth-protocol-rp` includes UserInfo client helpers. | Authentication API exposes `/userinfo`; SDKs retrieve user profile. | OIDC UserInfo endpoint is in OIDC reference and SDK flows. | Realm UserInfo endpoint in OIDC endpoint layer. | OAuth/UserInfo endpoint appears in OAuth endpoint docs. | `/oauth2/userInfo` exposed by user-pool OAuth endpoints. |
| Resource Server / Protected Resource | Consumer-side [`tigrbl-authz-resource-server`](../pkgs/30-capabilities/tigrbl-authz-resource-server/README.md) defines verifier contracts; [`resource-validation-api`](../pkgs/70-apis/tigrbl-auth-api-resource-validation/README.md) exposes metadata/JWKS/introspection for protected APIs. | Auth0 models APIs/resource servers in Management API and validates JWTs through issuer/JWKS; Auth0 FGA is adjacent authorization product. | Okta API Access Management models custom authorization servers/scopes; JWT verifier libraries support protected APIs. | Keycloak Authorization Services and policy enforcer cover protected resources, scopes, permissions, and UMA patterns. | Applications, JWT API, scopes, roles, and introspection support protected APIs. | User-pool resource servers/scopes and JWT validation support API protection; Identity Pools vend AWS credentials for AWS resources. |
| Token introspection / validation | `resource-validation-api` exposes `/introspect`; resource-server package handles local verification contracts. | Token validation primarily via JWT/JWKS; introspection-style behavior is less central than local JWT validation for APIs. | Introspection endpoint exists for Okta authorization servers; JWT verifier libraries support local validation. | Realm OIDC layer supports introspection; policy enforcer/resource server adapters can validate/authorize. | JWT API includes token introspection/validation features. | Cognito commonly uses local JWT validation with JWKS; no general OAuth introspection endpoint is a central user-pool product surface. |
| Client registration / application lifecycle | `public-api` includes dynamic registration endpoints; [`developer-api`](../pkgs/70-apis/tigrbl-auth-api-developer/README.md) owns tenant developer app/client management. | Management API clients/applications cover app lifecycle; dynamic client registration is not the primary advertised control plane. | Applications API and dynamic client registration/open standards support app lifecycle; service apps cover M2M. | Client Registration Service plus Admin REST client resources. | Application API covers client/application lifecycle; OAuth scopes and registrations are separate resources. | App clients are managed through Cognito APIs/console; no broad public dynamic client registration product surface. |
| Logout / session management | `public-api` exposes logout; `tigrbl-auth-protocol-rp` includes RP-initiated logout helpers; OIDC logout standards are represented in `tigrbl-auth-protocol-oidc`. | Logout via Authentication API/Universal Login and SDK logout helpers. | Sign-out/logout supported through OIDC endpoints and SDK/widget behavior. | Realm logout endpoint, session management, admin session controls, and adapters. | OAuth logout/hosted login session behavior and refresh-token/session APIs. | `/logout` endpoint through hosted UI/managed login; app clients configure sign-out URLs. |
| Upstream IdP / federation subsystem | Federation/IdP product surface is currently tracked as a gap/underframed in product matrices; primitives exist but frontdoor product surface is not first-class yet. | Connections/identity providers cover social, enterprise, SAML, OIDC, database, and passwordless connections. | Identity Providers API/concepts cover social, SAML, OIDC, routing, and account linking. | Identity brokering is a core realm feature with IdP mappers and first-login flows. | Identity Provider APIs cover external IdPs and SAML/OIDC/social integrations. | User pools support social, SAML, and OIDC federation providers. |
| Non-human / M2M / service-account client | First-class M2M lane was newly framed in ADR/SPEC work; service/workload identities and service keys are exposed through [`service-admin-api`](../pkgs/70-apis/tigrbl-auth-api-service-admin/README.md), while token issuance is through `public-api`. | Machine-to-machine applications use client credentials against Authentication API; clients are managed in Management API. | OAuth for Okta service apps and client credentials support M2M, especially for Okta management APIs. | Service accounts on confidential clients use client credentials/service-account roles. | Applications, API keys, and client credentials support service-to-service use. | App clients with client secrets plus resource servers/scopes support client credentials for M2M. |
| Admin / control-plane actor | Split frontdoors: platform admin, tenant admin, developer, service admin; target architecture frames these as Management API slices rather than separate product universes. | Dashboard and Management API v2 are the main control plane. | Admin Console and Management API are the main control plane. | Admin Console and Admin REST API are the main control plane. | Admin UI and API catalog are the main control plane; Tenant Manager supports delegated tenant admin. | AWS Console, Cognito APIs, IAM, and CloudTrail form the control plane. |
| SDK / UIX edge products | UIX apps exist for public/platform/tenant/developer/service-admin, plus RP/browser package; maturity varies by surface. | SPA, server, mobile, Management SDKs, Universal Login, Dashboard. | Auth JS, framework/mobile SDKs, Sign-In Widget, hosted sign-in, Admin/End-User dashboards. | JavaScript adapter, admin client, policy enforcer, Admin Console, Account Console, themes. | Client libraries, React/Angular/Vue SDKs, Hosted Login, Admin UI, themes. | AWS SDKs, Amplify Auth, Amplify UI Authenticator, Managed Login, AWS Console. |

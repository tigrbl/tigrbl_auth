Research date: 2026-05-28

## Interpretation

The vendor pattern is consistent: MFA is treated as an authentication product surface, not just a credential kind. Mature products expose:

| Product concern | Mature vendor behavior |
| --- | --- |
| Factor inventory | User-visible and admin-visible enrolled factors with type, status, verification state, and lifecycle. |
| Enrollment | Hosted login or API-driven setup with TOTP QR/secret, WebAuthn ceremonies, phone/email verification, and recovery-code display. |
| Challenge | Session-bound challenges with expiry, attempt limits, replay prevention, and a way to resume the original login or step-up operation. |
| Policy | Tenant/app/user/group/risk rules deciding when MFA is required and which factors satisfy assurance. |
| Recovery | Recovery codes, admin reset, self-service recovery, and safe lost-device workflows. |
| Audit | Factor enrollment, challenge, success, failure, reset, recovery, revocation, and policy-change events. |
| Claims | `amr` and `acr` or equivalent assurance signals for downstream applications and APIs. |

For `tigrbl_auth`, the clean product split is:

| Layer | Recommended owner |
| --- | --- |
| Factor proof material | `tigrbl-authn-credentials` |
| MFA requirement decisions | `tigrbl-authz-policy` |
| Challenge/session continuation | `tigrbl-auth-backend-app-public`, `tigrbl-auth-protocol-oauth`, `tigrbl-auth-protocol-oidc` |
| Factor and challenge storage | `tigrbl-identity-storage` |
| Tenant policy and support reset | `tigrbl-auth-backend-app-tenant-admin` |
| Hosted login MFA UI | `@tigrbl-auth/public-uix` |
| Assurance claims | `tigrbl-identity-jose`, `tigrbl-auth-protocol-oidc` |

Recommended near-term stance:

1. Start with TOTP plus recovery codes as the first production MFA slice.
2. Add WebAuthn/passkeys as the phishing-resistant target after baseline challenge/session infrastructure exists.
3. Treat SMS/email as optional lower-assurance reach factors, not primary security posture.
4. Do not expose admin factor reset without audit and tenant-boundary tests.
5. Do not claim MFA support until public login, policy, storage, UI, audit, and token claims work together.
